## 一、模块定位

本模块的核心目标是：**在给定课程上下文约束下，基于认知路径模型生成本节课的"认知路径实例"，并将其展开为可执行的学生活动结构，供课堂直接使用。**

---

## 二、整体结构层级

系统采用六层嵌套结构，从宏观约束到微观执行依次收敛：

| 层级 | 名称 | 定位 |
|------|------|------|
| 1 | **Context**（课程上下文） | 定义本节课的生成边界与约束条件 |
| 2 | **Model**（认知路径模型） | 定义阶段逻辑，由学校提供或使用系统默认 |
| 3 | **Path Instance**（路径实例） | 将抽象模型转化为本节课的具体结构 |
| 4 | **Node**（认知推进节点） | 课堂认知推进的核心单位 |
| 5 | **ActivityTemplate**（活动模板） | 可复用的活动结构模板，介于节点与活动之间 |
| 6 | **Activity**（教学活动） | 面向学生的可执行任务单元 |

> 🔧 **[架构扩展]** 原文档为五层结构。本次在 Node 与 Activity 之间新增 **ActivityTemplate** 层（对应建议三），使层级扩展为六层。

生成关系为：`Context + Model → Path Instance → Node → (ActivityTemplate) → Activity`

---

## 三、课程上下文模型（Context）

Context 用于限定本节课的生成边界，其字段直接影响下游各层的生成逻辑。

### 3.1 字段定义

**教学主题（Topic）**

本节课聚焦的核心知识领域。示例：K-means 聚类算法、人工智能的三次浪潮、图像识别中的特征提取、神经网络基本结构。

> 影响：`Node.learning_goal` 的语义范围。

---

**核心问题（Core Question）**

驱动整节课认知推进的关键问题。示例：不同 K 值如何影响聚类效果？为什么人工智能经历多次低谷？为什么特征会影响模型表现？

> 影响：`Node.node_type`、`Node.learning_goal`、`Node.logic_relation`。

---

**核心概念（Concepts）**

本节课必须被理解的关键概念集合，数量限定为 3–6 个。示例：聚类、K 值、聚类中心、特征学习、无监督学习。

> 影响：`Node.learning_goal`、`Activity.task_description`。

---

**核心素养（Competency）**

本节课应促成的认知水平变化，对应学科核心素养框架。

> 影响：`Node.node_type`、`Node.logic_relation`。

---

**目标能力（Ability）**

学生本节课应获得的具体能力表述，需提供量规说明，并能归入课程核心素养以便后续评价。示例：能比较不同参数设置结果，能解释 K 值对结果的影响；能评价当前 AI 发展趋势。

> 影响：`Node.evidence_type`、`Activity.deliverable_type`、`Activity.evaluation_rubric`。

---

**课时长度（Duration）**

课堂可用时间，固定选项：35 分钟 / 40 分钟 / 60 分钟 / 90 分钟 / 120 分钟。

> 影响：`PathInstance.stage` 数量、`Node` 数量、`Node.estimated_time`、`Activity` 数量。

---

**课程类型标签（Type）**

课程主要认知形式，固定选项：**概念建构**（讲授式）/ **自主探究**（探究式）。

> 影响：`Node.node_type`、`Activity.activity_type`、`Node.teacher_support_behavior`。

---

**学生学段（Student_grade）**

课程主要面向的学生范围，固定选项：小低（1–3 年级）/ 小高（4–6 年级）/ 初中 / 高中 / 中职 / 高职 / 大学本科。

---

**学生水平（Student_level）**

> 🔧 **[架构扩展]** 原文档标注为"暂不开发，仅作占位"。本次扩展其战略定义（对应建议五）：

`Student_level` 是系统从"课程设计工具"升级为"自适应教学系统"的关键入口。当前阶段前端不展示该字段，但数据模型层面需完成以下预留工作：

- 在 Context 数据结构中保留 `student_level` 字段，允许为空；
- 新增关联字段 `student_profile_id`，用于未来接入学生画像系统；
- 后台在每次生成 Path Instance 时，记录当前 `student_level` 值（即使为空），为未来个性化生成能力积累训练数据。

## 四、认知路径模型（Model）

路径模型由学校提供或使用系统默认模型。**模型只定义阶段逻辑，不规定节点内容。**

### 4.1 字段定义

| 字段 | 说明 |
|------|------|
| `model_id` | 模型唯一标识，所有后续 `stage_instances` 必须符合该模型结构 |
| `model_name` | 模型名称，如：认知修订模型、问题解决模型 |
| `stage_definitions[]` | 模型所包含的阶段结构列表 |
| `suggested_node_range` | 该模型在标准课时下建议的节点数量范围 |
| `logic_relation_set[]` | **⚡ 新增** 该模型支持的节点间关系枚举集合，详见 4.2 |

> 🔧 **[架构扩展]** 原文档中 `logic_relation` 为全局固定值（建立 / 对比 / 推翻 / 应用），覆盖范围仅适用于认知修订模型。本次新增 `logic_relation_set[]` 字段（对应建议二），将枚举集合的定义权下放至 Model 层，不同模型可声明各自支持的关系类型，实现模型可扩展性。

### 4.2 各模型 logic_relation_set 示例

| 模型 | 支持的 logic_relation 枚举集合 |
|------|-------------------------------|
| 认知修订模型 | 建立、对比、推翻、应用 |
| 问题解决模型 | 建立、分解、验证、迁移、应用 |
| 项目制学习模型 | 建立、分解、整合、迭代、展示 |
| 探究学习模型 | 建立、假设、验证、修正、推广 |

> Node 层的 `logic_relation` 字段取值必须来自其所属 Path Instance 对应 Model 的 `logic_relation_set[]`，不得使用集合外的值。系统在生成和校验时应强制执行此约束。

**示例：** 认知修订模型包含 7 个阶段——明确认知对象、观察认知对象、提出个体断言、小组交流与初步修订、班级共议与再修订、持续修订并记录过程、概括提升与提炼认知。在 40 分钟课堂下，建议节点数量为 5–7。

---

## 五、认知路径实例（Path Instance）

路径实例是某节课在某一模型下的具体结构实现，将抽象模型转化为可执行结构。

### 5.1 Path Instance 字段

| 字段 | 说明 |
|------|------|
| `model_id` | 所使用的认知路径模型标识 |
| `duration` | 本节课实际可用时间，应与 `Context.duration` 一致 |
| `stage_instances[]` | 本节课在所选模型下生成的阶段实例列表 |
| `status` | 编辑与版本控制状态，**⚡ 完整生命周期**，详见 5.2 |

### 5.2 status 生命周期状态机

> 🔧 **[架构扩展]** 原文档 `status` 仅有草稿 / 确认 / 发布三个状态，缺少课堂执行后的反馈闭环（对应建议四）。本次扩展为完整的五状态生命周期：

```
草稿 → 确认 → 发布 → 已使用 → 已归档
```

| 状态 | 触发条件 | 说明 |
|------|----------|------|
| **草稿** | 创建时默认 | 可自由编辑所有字段 |
| **确认** | 教师手动确认 | 结构锁定，仅允许微调活动内容 |
| **发布** | 教师发布 | 对学生可见，课堂执行中 |
| **已使用** | 课堂结束后标记 | 允许教师添加执行备注；系统记录实际执行时间与设计时间的差异 |
| **已归档** | 版本迭代或手动归档 | 只读，保留历史版本供对比 |

"已使用"状态下新增字段：

| 字段 | 说明 |
|------|------|
| `execution_notes` | 教师课后执行备注（自由文本） |
| `actual_time_used` | 各节点实际使用时间记录 |
| `revision_trigger` | 触发下一版本修订的原因标注 |

### 5.3 Stage Instance 字段

Stage Instance 由 `Model.stage_definitions[]` 确定，包含以下字段：

| 字段 | 说明 |
|------|------|
| `stage_type` | 阶段类型，继承自模型定义 |
| `stage_order` | 阶段顺序编号 |
| `node_list[]` | 该阶段下的节点列表 |

## 六、认知推进节点模型（Node）

**节点是本系统的核心单位**，承载认知推进的逻辑与目标，不写具体题目或详细任务文本。

### 6.1 字段定义

| 字段 | 说明 | 影响来源 |
|------|------|----------|
| `node_id` | 节点唯一标识 | — |
| `node_type` | 认知行为类型，如：概念理解、实验探究、比较分析、抽象总结、迁移应用 | `Type`、`Core Question`、`Competency` |
| `learning_goal` | 本节点认知目标，如：理解 K 值改变会影响聚类边界 | `Topic`、`Core Question`、`Concepts` |
| `student_behavior` | 学生行为描述，如：修改 K 值并观察结果 | `Student_grade`、`Type` |
| `evidence_type` | 判断达成的证据形式，如：输出不同聚类图、提交分析文字 | `Student_grade`、`Ability` |
| `teacher_support_behavior` | 教师行为描述，如：引导学生分析第一次低谷的原因 | — |
| `estimated_time` | 节点持续时间（分钟） | `Duration`、节点数量 |
| `logic_relation` | 与前节点的认知关系，**⚡ 取值范围由所属 Model 的 `logic_relation_set[]` 决定** | `Core Question`、`Competency`、`Model` |

> 🔧 **[架构扩展]** `logic_relation` 的取值约束已从全局固定枚举改为模型绑定枚举（见第四章 4.2），此处字段定义同步更新。

---

## 七、活动模板层（ActivityTemplate）

> 🔧 **[架构扩展]** 本章为新增章节（对应建议三）。原文档中活动模板仅在 Activity 章节末尾以一句话提及，缺乏结构定义。本次将其提升为独立的数据层，作为 Node 与 Activity 之间的中间层。

ActivityTemplate 是可复用的活动结构模板，由教师社区沉淀或系统预置。模板携带预设的 `behavior_sequence`，可显著降低活动生成成本，并在教师间形成优质设计的流通机制。

### 7.1 ActivityTemplate 字段定义

| 字段 | 说明 |
|------|------|
| `template_id` | 模板唯一标识 |
| `template_name` | 模板名称，如：参数实验型、对比分析型、证据推理型、概念归纳型 |
| `activity_type` | 对应的活动类型 |
| `applicable_node_types[]` | 适用的节点类型列表，如：`[实验探究, 变量控制]` |
| `behavior_sequence[]` | 预设的行为序列，结构与 Activity.behavior_sequence 一致 |
| `source` | 模板来源，固定值：系统预置 / 教师创建 / 社区推荐 |
| `usage_count` | 被引用次数，用于社区排序 |

### 7.2 模板与 Activity 的关联规则

Activity 新增 `template_id` 字段，用于关联所使用的模板。关联为可选项：

- 若 `template_id` 存在，系统以模板的 `behavior_sequence` 为初始值，允许教师在此基础上修改；
- 若 `template_id` 为空，表示该活动为完全自定义，不依赖任何模板。

教师在使用模板后，可选择将修改后的版本保存为新模板，触发社区沉淀流程。

## 八、教学活动模型（Activity）

活动是执行层，面向学生的可操作任务单元。

**结构原则：**
- 一个节点可包含多个活动；
- 一个活动必须归属于一个节点；
- 学生活动为系统重点呈现对象。

### 8.1 Activity 字段定义

| 字段 | 说明 |
|------|------|
| `activity_id` | 活动唯一标识 |
| `node_id` | 所属节点标识 |
| `template_id` | **⚡ 新增** 关联的活动模板标识，可为空 |
| `activity_type` | 活动类型，如：操作实验、讨论表达、阅读理解、参数测试、单项选择、单项填空、多项填空 |
| `task_description` | 具体任务文本，如：将 K 值改为 2 并截图结果 |
| `tool_required` | 工具要求，固定选项：AI 绘图 / Python 编辑器 / 手势模型训练等实验中心入口；推荐设计交互网页 |
| `deliverable_type` | 提交类型，固定选项：代码 / 图片 / 网址 / 文本 / 选择结果 |
| `evaluation_mode` | 评价类型，如：自动检测 / 规则检测 / 人工评价 |
| `evaluation_rubric` | 评价标准描述 |
| `estimated_time` | 活动持续时间（分钟） |

> 🔧 **[架构扩展]** 新增 `template_id` 字段，用于关联第七章定义的 ActivityTemplate 层（对应建议三）。

## 九、活动生成约束（Execution Constraints）

Execution Constraints 作为活动生成的硬性约束条件，用于过滤不可执行的活动方案。

### 9.1 工具环境（Tool Environment）

| 字段 | 说明 |
|------|------|
| `device_type` | 无设备 / 平板 / 电脑 |
| `software_environment` | 浏览器 / Python 环境 / AI 平台 |

> 影响：`Activity.tool_required`。示例：K-means 实验必须有电脑环境。

### 9.2 材料复杂度（Material Complexity）

| 字段 | 说明 |
|------|------|
| `material_level` | 无材料 / 简单材料（纸笔）/ 需提前准备 / 需购买 |

> 规则：系统默认只生成前两类，否则老师不会执行。

### 9.3 时间准备成本（Preparation Cost）

| 字段 | 说明 |
|------|------|
| `prep_time` | 无准备 / 课前 5 分钟 / 课前 30 分钟 |

> 规则：若 `prep_time > 30min`，系统需向教师发出提示。

### 9.4 教室空间结构（Space Requirement）

| 字段 | 说明 |
|------|------|
| `space_mode` | 个体 / 小组 / 全班 |

> 示例：若班级座位无法移动，系统不建议生成需要小组讨论的活动。

### 9.5 课堂管理复杂度（Management Complexity）

| 字段 | 说明 |
|------|------|
| `management_level` | 低（安静操作）/ 中（讨论）/ 高（走动/展示） |

> 系统默认限制为"中"。

### 9.6 预算成本（Budget）

| 字段 | 说明 |
|------|------|
| `cost_level` | 0（无成本）/ 低 / 需预算 |

> 系统默认：`cost_level = 0`。

## 十、学习行为模型（Learning Behavior）

Learning Behavior 作为活动生成的行为约束，与 Execution Constraints 相互验证，确保活动的可执行性与认知合理性。

### 10.1 behavior_sequence[] 字段

每个活动包含一组有序的行为序列，每条行为包含以下字段：

| 字段 | 说明 |
|------|------|
| `behavior_type` | 行为类型，见 10.2 最小集合 |
| `behavior_target` | 行为对象，如：数据结果、算法输出、历史材料 |
| `behavior_tool` | 使用工具，如：Python 环境、实验网页、教材材料 |
| `behavior_output` | 行为产出，如：截图、填写表格、文本总结、选择答案 |
| `estimated_time` | 行为持续时间（分钟） |

`Node.node_type` 影响 `behavior_sequence` 的组合逻辑。

### 10.2 Learning Behavior 最小集合（按课堂使用频率排序）

**第一层：基础信息处理行为**（最高频，几乎所有课程使用）

阅读材料、观察现象、听取讲解、标记重点、提取信息、记录内容。

适用场景：概念学习、案例分析、资料阅读（如 AI 发展史课程）。

**第二层：表达与反馈行为**（课堂互动最常见，直接连接评价系统）

回答问题、完成选择、填写表格、撰写简短解释、提交答案、展示结果。

**第三层：思维加工行为**（认知推进的核心行为）

比较差异、分类整理、总结规律、提出观点、解释原因、提出假设。

适用场景：K-means 参数实验、AI 三次浪潮分析、数据分析类课程。

**第四层：协作行为**（课堂组织行为，影响课堂管理结构）

小组讨论、共同整理、互相评价、集体汇报。

**第五层：操作行为**（实践课程核心，人工智能与信息技术课程高频使用）

运行程序、修改参数、观察输出、调试程序、绘制图表、构建简单模型。

### 10.3 行为-约束兼容矩阵

> 🔧 **[架构扩展]** 原文档中 Learning Behavior 与 Execution Constraints 的验证关系为隐式描述（对应建议一）。本次将验证规则显式化，形成兼容矩阵。系统在生成活动时，应根据当前 Execution Constraints 的实际值，自动过滤矩阵中标注为"❌ 不兼容"的行为类型，无需人工校验。

| 行为类型（behavior_type） | 最低 device_type | 最低 space_mode | 最低 management_level |
|--------------------------|-----------------|----------------|----------------------|
| 阅读材料 / 听取讲解 / 标记重点 | 无设备 | 个体 | 低 |

## 十一、生成流程逻辑（强制顺序）

系统采用强制线性生成流程，各步骤不可跳过：

1. 确认课程上下文（Context）
2. 选择认知路径模型（Model）
3. 生成认知节点草图（Path Instance + Node）
4. 教师确认 / 编辑节点
5. 生成教学活动（Activity，可选择 ActivityTemplate 加速）
6. 预览与发布

---

## 十二、结构原则

- 路径当前阶段为**线性结构**；
- 节点数量由课时长度建议，不强制固定；
- 模型可替换，节点结构不可替换；
- **结构先于内容生成**；
- `logic_relation` 取值由所属 Model 的枚举集合约束，不同模型可扩展各自的关系类型。

## 附录：架构分析

### 你在做什么

你正在构建一套**面向教师的结构化教学设计生成系统**，其本质是一个**约束驱动的内容生成引擎**。

系统的核心逻辑是：用 Context 作为语义约束集，用 Model 作为结构骨架，通过两者的交叉推导出一个符合认知规律的 Path Instance，再将其逐层展开为可执行的 Activity。整个架构的设计哲学是**"结构先于内容"**——先确定认知推进的骨架，再填充具体的教学行为，这与传统教案设计"先写内容再考虑结构"的习惯形成了根本性的反转。

这不是一个简单的表单填写工具，也不是一个内容推荐系统，而是一套试图将**教学设计的专业知识编码为系统约束**的知识工程产品。它的目标用户（教师）通常缺乏系统化教学设计训练，而这套架构试图通过结构化引导，让普通教师也能产出具有认知逻辑的教学方案。
